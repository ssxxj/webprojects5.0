## 三、关系图/关系表三档说明

1. 未提交：记 0。
2. 已提交但主要停留在工具解释、概念堆叠、没有体现关系链：按部分完成扣分。
3. 已提交且明确体现关系、流程、因果、控制点与下一步：满分。

## 四、证据 -> 结论映射提醒

1. 线索不等于证据。
2. 第三方索引不等于你本人实时做出的主动探测。
3. 开放端口或服务不等于漏洞已经成立。
4. 局部请求与响应不等于更大结论自动成立。

## 五、关键字段示例

1. Web三层架构与数据流图：浏览器、服务器、数据库三层角色；输入从哪里进入；结果从哪里返回；至少1个安全关注点
2. HTTP请求与响应基础记录：请求方法；URL或路径；参数位置；状态码或返回结果
3. Cookie与Session状态维持机制说明：登录前后状态变化；至少1个Cookie或Session线索；为什么登录后还能保持状态；Cookie与Session分工
4. 输入—处理—输出—执行—身份框架应用：至少1个简单页面场景；输入点与输出点；至少2个风险点或控制点；统一分析框架解释

## 六、关系图/关系表正反例提示

1. 错误写法：只列工具、术语、概念，或只列结果，不写关系。
2. 合格写法：至少写出“上一层结果如何导向下一步”。
3. 较好写法：写出“现象/输入 -> 处理/机制 -> 结果 -> 控制点 -> 下一步”。
4. 与 系统链路图/关系图 相关的结论必须体现“能说明什么 / 不能替代什么 / 下一步为什么是这个”。

## 七、红线规则

1. `未脱敏`：截图中敏感信息未脱敏，处理方式：`lt_60`。
2. `超出授权边界`：对未授权目标进行观察、测试或抓包，处理方式：`lt_60`。
3. `证据伪造或不可追溯`：截图、字段或过程明显不真实，无法支撑作业结论，处理方式：`lt_60`。
4. `高度雷同或抄袭代做`：与他人作业高度雷同，或存在抄袭、代做情形，处理方式：`lt_60`。

## 八、专业校验重点

1. 是否区分 HTTP/HTTPS 传输保护与业务安全
2. 是否区分 Cookie 与 Session 的职责
3. 是否能用证据解释状态维持，而不是只背术语
4. 是否避免将 IP 地址绝对化地写成主要身份识别依据
5. 是否对授权边界和脱敏要求给出负责任表达
