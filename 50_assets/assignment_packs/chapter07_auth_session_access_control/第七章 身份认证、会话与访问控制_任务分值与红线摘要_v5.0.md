## 五、关系图/关系表三档说明

1. 未提交：记 0。
2. 已提交但主要停留在工具解释、概念堆叠、没有体现关系链：按部分完成扣分。
3. 已提交且明确体现关系、流程、因果、控制点与下一步：满分。

## 六、证据 -> 结论映射提醒

1. 线索不等于证据。
2. 第三方索引不等于你本人实时做出的主动探测。
3. 开放端口或服务不等于漏洞已经成立。
4. 局部请求与响应不等于更大结论自动成立。

## 七、关键字段示例

1. 认证、会话与访问控制系统模型：认证负责什么；会话负责什么；访问控制负责什么；至少1个从登录到敏感操作的流程场景
2. Brute Force Low/Medium 观察与分析：至少1个暴力破解实验现象；Low 与 Medium 至少1项差异；说明为什么属于认证成本问题；至少2个控制点
3. Weak Session IDs 机制说明：会话值出现位置；可预测性或随机性线索；为什么会破坏持续信任；认证与会话的差异
4. CSRF 现象与来源校验分析：CSRF 的基础现象或流程；为什么不是密码泄露问题；会话可信不等于请求可信；至少1种来源校验思路
5. 访问控制与身份安全加固清单：访问控制不能由是否登录替代；至少4条加固建议；说明建议对应身份、状态、来源或权限哪一环；至少1个最小权限思路

## 八、关系图/关系表正反例提示

1. 错误写法：只列工具、术语、概念，或只列结果，不写关系。
2. 合格写法：至少写出“上一层结果如何导向下一步”。
3. 较好写法：写出“现象/输入 -> 处理/机制 -> 结果 -> 控制点 -> 下一步”。
4. 与 认证—会话—请求来源—访问控制流程图/关系图 相关的结论必须体现“能说明什么 / 不能替代什么 / 下一步为什么是这个”。

## 九、红线规则

1. `未脱敏`：截图、账号、Cookie、SessionID 或其他敏感信息未按要求脱敏，处理方式：`lt_60`。
2. `超出授权边界`：对未授权真实目标进行认证、会话或来源相关测试，处理方式：`lt_60`。
3. `证据伪造或不可追溯`：现象截图、实验过程或说明明显无法支撑结论，处理方式：`lt_60`。
4. `高度雷同或抄袭代做`：与他人作业高度雷同，或存在抄袭、代做情形，处理方式：`lt_60`。

## 十、专业校验重点

1. 是否区分认证、会话、来源校验和访问控制四个环节
2. 是否说明暴力破解更接近认证成本设计问题，而不只是弱口令问题
3. 是否说明会话值的随机性与不可预测性是持续信任的基础
4. 是否说明 CSRF 的根因是来源与意图失配，而不是密码泄露
5. 是否说明“已登录”不能替代访问控制，并对授权边界给出负责任表达
