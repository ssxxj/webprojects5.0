## 五、关系图/关系表三档说明

1. 未提交：记 0。
2. 已提交但主要停留在工具解释、概念堆叠、没有体现关系链：按部分完成扣分。
3. 已提交且明确体现关系、流程、因果、控制点与下一步：满分。

## 六、证据 -> 结论映射提醒

1. 线索不等于证据。
2. 第三方索引不等于你本人实时做出的主动探测。
3. 开放端口或服务不等于漏洞已经成立。
4. 局部请求与响应不等于更大结论自动成立。

## 七、关键字段示例

1. 应用层到系统层执行链说明：应用层与系统层角色；输入如何进入系统调用链；为什么属于跨层风险；至少1个业务场景
2. DVWA Command Injection Low 现象记录：实验页面或目标；输入与结果差异；系统执行迹象；为什么不只是页面回显变化
3. Low/Medium 控制差异与命令结构分析：Low 与 Medium 至少2项差异；说明这些差异控制了输入表面、执行结构还是后果范围；为什么禁止几个符号通常不够；与 SQL 注入的至少1个共性
4. 改进方案与防护清单：参数分离或替代安全接口；最小权限或最小化暴露面；结果处理或边界限制；至少4条改进建议并说明对应控制点

## 八、关系图/关系表正反例提示

1. 错误写法：只列工具、术语、概念，或只列结果，不写关系。
2. 合格写法：至少写出“上一层结果如何导向下一步”。
3. 较好写法：写出“现象/输入 -> 处理/机制 -> 结果 -> 控制点 -> 下一步”。
4. 与 应用层到系统层风险链路图/防护关系图 相关的结论必须体现“能说明什么 / 不能替代什么 / 下一步为什么是这个”。

## 九、红线规则

1. `未脱敏`：截图、命令输出、账号、路径或其他敏感信息未按要求脱敏，处理方式：`lt_60`。
2. `超出授权边界`：对未授权真实目标进行命令执行测试或其他验证，处理方式：`lt_60`。
3. `证据伪造或不可追溯`：现象截图、结果说明或实验过程明显不一致，处理方式：`lt_60`。
4. `高度雷同或抄袭代做`：与他人作业高度雷同，或存在抄袭、代做情形，处理方式：`lt_60`。

## 十、专业校验重点

1. 是否说明命令执行的关键在于输入进入命令结构，而不是记住某条命令
2. 是否区分应用层行为变化与系统层执行变化
3. 是否说明黑名单限制、参数分离、替代接口和最小权限的控制层次
4. 是否把命令执行与 SQL 注入放回“数据进入执行结构”的共同框架
5. 是否对跨层风险、授权边界和后果范围给出负责任表达
