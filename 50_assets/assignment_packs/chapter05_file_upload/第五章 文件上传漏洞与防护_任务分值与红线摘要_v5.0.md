## 一、章节主线

从上传链路观察，到控制链完整性设计

## 三、关系图/关系表三档说明

1. 未提交：记 0。
2. 已提交但主要停留在工具解释、概念堆叠、没有体现关系链：按部分完成扣分。
3. 已提交且明确体现关系、流程、因果、控制点与下一步：满分。

## 四、证据 -> 结论映射提醒

1. 线索不等于证据。
2. 第三方索引不等于你本人实时做出的主动探测。
3. 开放端口或服务不等于漏洞已经成立。
4. 局部请求与响应不等于更大结论自动成立。

## 五、关键字段示例

1. 文件上传业务流程与风险链说明：上传流程的主要阶段；至少4个控制点或风险点；为什么上传不是单一步骤；至少1个业务场景
2. DVWA File Upload Low 现象记录：实验页面或目标；上传结果或保存证据；访问路径或后续处理线索；至少3个明显缺失的控制点
3. Low/Medium 控制差异比较：Low 与 Medium 至少2项差异；这些差异控制了哪一环；为什么单点检查不等于完整防护；至少1个残余风险
4. 上传防护清单与业务方案设计：扩展名、MIME、内容检查等验证措施；重命名、隔离存储、禁执行或访问控制；至少4条防护建议；说明每条措施对应哪一环

## 六、关系图/关系表正反例提示

1. 错误写法：只列工具、术语、概念，或只列结果，不写关系。
2. 合格写法：至少写出“上一层结果如何导向下一步”。
3. 较好写法：写出“现象/输入 -> 处理/机制 -> 结果 -> 控制点 -> 下一步”。
4. 与 文件上传链路图/风险控制关系图 相关的结论必须体现“能说明什么 / 不能替代什么 / 下一步为什么是这个”。

## 七、红线规则

1. `未脱敏`：截图、文件名、路径、账号或其他敏感信息未按要求脱敏，处理方式：`lt_60`。
2. `超出授权边界`：对未授权真实目标进行上传测试或其他验证，处理方式：`lt_60`。
3. `证据伪造或不可追溯`：现象截图、上传记录或说明与实验过程明显不一致，处理方式：`lt_60`。
4. `高度雷同或抄袭代做`：与他人作业高度雷同，或存在抄袭、代做情形，处理方式：`lt_60`。

## 八、专业校验重点

1. 是否说明上传问题本质上是链路控制问题，而不只是文件进来这一步
2. 是否区分扩展名检查、MIME 检查、内容检查、存储隔离和执行隔离
3. 是否避免把“上传成功”直接当成漏洞成立或防护失效的全部依据
4. 是否说明前端限制不能替代服务端控制链
5. 是否对授权边界、公开访问风险和执行风险给出负责任表达
