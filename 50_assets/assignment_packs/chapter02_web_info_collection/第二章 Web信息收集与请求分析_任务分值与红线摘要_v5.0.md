## 三、关系图/关系表三档说明

1. 未提交：记 0。
2. 已提交但主要停留在工具解释、概念堆叠、没有体现关系链：按部分完成扣分。
3. 已提交且明确体现关系、流程、因果、控制点与下一步：满分。

## 四、证据 -> 结论映射提醒

1. 线索不等于证据。
2. 第三方索引不等于你本人实时做出的主动探测。
3. 开放端口或服务不等于漏洞已经成立。
4. 局部请求与响应不等于更大结论自动成立。

## 五、关键字段示例

1. Google 公开页面线索记录：查询对象；查询语句；至少一种搜索语法；至少1条公开线索
2. Whois 域名身份信息记录：查询对象；注册商或注册时间等字段；Name Server或其他关键字段；能说明什么
3. Shodan 服务暴露线索记录：查询条件；IP或主机标识；至少1个端口；至少1个服务线索
4. 社交工程风险场景分析：案例来源；至少2个泄露点；后果分析；至少3条防范建议
5. Burp 请求证据观察：抓包目标；Method；URL；参数位置
6. Nmap 授权主动探测记录：扫描目标；扫描命令；至少1个端口；状态与服务

## 六、关系图/关系表正反例提示

1. 错误写法：只列工具、术语、概念，或只列结果，不写关系。
2. 合格写法：至少写出“上一层结果如何导向下一步”。
3. 较好写法：写出“现象/输入 -> 处理/机制 -> 结果 -> 控制点 -> 下一步”。
4. 与 六层信息综合关系图/表 相关的结论必须体现“能说明什么 / 不能替代什么 / 下一步为什么是这个”。

## 七、红线规则

1. `未脱敏`：截图或文字中保留敏感信息、Session、Token等未脱敏内容，处理方式：`lt_60`。
2. `超出授权边界`：对未授权真实目标进行抓包、扫描或其他测试，处理方式：`lt_60`。
3. `证据伪造或不可追溯`：提交内容明显无法支撑结论，或截图、记录与正文严重不一致，处理方式：`lt_60`。
4. `高度雷同或抄袭代做`：与他人作业高度雷同，或存在抄袭、代做情形，处理方式：`lt_60`。

## 八、专业校验重点

1. 是否区分线索、证据和主动探测
2. 是否明确 Shodan 属于第三方索引而非本人实时探测
3. 是否说明 Burp 看到的是真实请求交互证据
4. 是否说明 Nmap 的开放端口不等于漏洞成立
5. 是否对授权边界、风险和下一步选择给出负责任表达
