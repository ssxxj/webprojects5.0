## 五、关系图/关系表三档说明

1. 未提交：记 0。
2. 已提交但主要停留在工具解释、概念堆叠、没有体现关系链：按部分完成扣分。
3. 已提交且明确体现关系、流程、因果、控制点与下一步：满分。

## 六、证据 -> 结论映射提醒

1. 线索不等于证据。
2. 第三方索引不等于你本人实时做出的主动探测。
3. 开放端口或服务不等于漏洞已经成立。
4. 局部请求与响应不等于更大结论自动成立。

## 七、关键字段示例

1. 数据库与查询条件基础说明：输入条件—数据库查询—返回结果；为什么页面功能与数据库有关；输入如何进入查询条件
2. DVWA SQL Injection Low 现象记录：实验目标或页面；普通输入与异常输入对比；返回差异；为什么不像正常查询变化
3. 机制分析与差异比较：输入—拼接—解析—执行—回显或行为差异链条；Low与Medium差异；普通SQL注入与盲注差异；根因是结构被污染而不是字符本身
4. SQL注入防护清单与解释：参数化查询；输入校验；错误处理；最小权限

## 八、关系图/关系表正反例提示

1. 错误写法：只列工具、术语、概念，或只列结果，不写关系。
2. 合格写法：至少写出“上一层结果如何导向下一步”。
3. 较好写法：写出“现象/输入 -> 处理/机制 -> 结果 -> 控制点 -> 下一步”。
4. 与 SQL注入因果链/防护关系图 相关的结论必须体现“能说明什么 / 不能替代什么 / 下一步为什么是这个”。

## 九、红线规则

1. `未脱敏`：截图、参数、账号或其他敏感信息未按要求脱敏，处理方式：`lt_60`。
2. `超出授权边界`：对未授权真实目标进行 SQL 注入测试或验证，处理方式：`lt_60`。
3. `证据伪造或不可追溯`：实验记录、现象截图或说明与实际过程严重不一致，处理方式：`lt_60`。
4. `高度雷同或抄袭代做`：与他人作业高度雷同，或存在抄袭、代做情形，处理方式：`lt_60`。

## 十、专业校验重点

1. 是否区分正常查询结果变化与查询逻辑被改变
2. 是否避免把 payload 或特殊字符本身写成根因
3. 是否说明盲注与普通 SQL 注入共享同一根因，只是证据形式不同
4. 是否区分过滤、参数化查询、错误处理和最小权限的控制层次
5. 是否对授权边界、复现实验和风险表达给出负责任说明
