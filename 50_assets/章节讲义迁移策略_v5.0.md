# 章节讲义迁移策略 v5.0

## 1. 定位

本文件定义 `projects5.0` 中章节讲义的迁移原则、迁移顺序、质量门槛与纳入条件。

5.0 的结论不是“整体复制 4.0 讲义”，而是：

`以 4.0 讲义为内容来源，以 5.0 profile / assignment pack / memory / 评估母规则为约束，逐章重建 5.0 讲义资产。`

## 2. Concept

### 2.1 章节讲义在 5.0 中是什么

在 `projects5.0` 中，章节讲义是连接“课程规则”和“课堂实施”的中间资产：

`课程母规则 -> 章节 profile -> assignment pack -> 章节讲义 -> 授课实施 -> 作业评估 -> memory 回写`

因此，讲义的职责是：

1. 承载章节主线
2. 把 assignment pack 中的执行要求解释成可教学内容
3. 把 profile 中的能力目标落实到单元结构
4. 为课堂 Observation / Decision / Evidence / Mechanism / Modification 提供教师可用脚本

### 2.2 章节讲义不是什么

1. 不是 `projects4.0` docx 的镜像副本
2. 不是 assignment pack 的简单改写
3. 不是先生成、后对齐规则的孤立文档
4. 不是只面向学生阅读的材料

## 3. Mechanism

### 3.1 上游约束

5.0 讲义迁移必须同时对齐四类上游资产：

1. `4.0 章节讲义原文`
   主要来源：
   - `projects4.0/50_assets/章节讲义/*-按OMDM修订稿.md`
   - `projects4.0/50_assets/章节讲义/*-学生预习版.md`
   - `projects4.0/20_skills/SK04_章节讲义加工器.md`
   - `projects4.0/20_skills/SK05_讲义质量双评器.md`
   - `projects4.0/20_skills/SK06_授课实施设计器.md`

2. `5.0 章节 profile`
   规定本章的：
   - 主线
   - 能力目标
   - 任务分值
   - 红线规则
   - 专业校验点

3. `5.0 assignment pack`
   规定本章的：
   - 作业执行面
   - 证据字段
   - 关系图/关系表要求
   - AI 输出审核要求
   - 教师验收字段

4. `5.0 memory`
   提供：
   - 高频误区
   - 正反样例
   - 诊断标签
   - 班级级反馈回流结果

### 3.2 迁移单位

5.0 讲义迁移的最小单位是 `单章`，不是整套八章。

每章正式迁移时，至少应形成四个资产：

1. `教师版章节讲义_v5.0.md`
2. `学生预习版章节讲义_v5.0.md`
3. `讲义迁移清单_v5.0.json`
4. `4.0讲义 -> 5.0讲义 -> 5.0作业包 对照表.md`

### 3.3 迁移原则

#### 原则 A：内容来源优先保留 4.0

5.0 讲义的知识内容、案例和章节主线，优先继承 4.0 OMDM 修订稿。

#### 原则 B：结构约束优先服从 5.0

只要 4.0 的某段内容与 5.0 的 profile、assignment pack、红线或 memory 相冲突，应按 5.0 重组，而不是机械保留原段落位置。

#### 原则 C：先教师版，再学生版

迁移顺序应为：

`教师版讲义 -> 质量复核 -> 学生预习版`

而不是先做学生版，再回头补教师版。

#### 原则 D：先单章跑通，再扩到整套

每章迁移后，至少应通过一次：

`讲义 -> 作业包 -> 评分 -> 教学反馈`

的闭环校验，再视为正式章节资产。

## 4. 推荐迁移顺序

当前推荐顺序不是按授课顺序，而是按“5.0 上游是否已稳定”排序。

### 第一批：立即可迁

1. 第二章 `Web信息收集与请求分析`
   原因：
   - 已有真实评分样本
   - 已有正式班级反馈
   - 已有 memory 回流
   - 已有回归后的 assignment pack

2. 第四章 `XSS漏洞与防护`
3. 第五章 `文件上传漏洞与防护`
4. 第六章 `命令执行漏洞与防护`
5. 第七章 `身份认证、会话与访问控制`
6. 第八章 `综合复盘与章节项目展示`
   原因：
   - assignment pack 已正式生成
   - 已有 4.0 -> 5.0 作业包对照
   - profile 已稳定

### 第二批：后续迁移

1. 第一章 `Web基础与HTTP请求响应`
2. 第三章 `SQL注入漏洞与防护`

原因：

这两章虽已有 profile，但目前还没有和第二章一样完成“真实班级反馈 -> memory 回写 -> 模板反推”的回归优势。

## 5. 章节迁移流程

### P0：冻结上游

迁移前先锁定本章以下文件：

1. chapter profile
2. assignment pack
3. 最新 memory 快照
4. 当前对照表

若其中任一项仍在频繁变动，不应生成正式讲义。

### P1：抽取 4.0 内容来源

从 4.0 中抽取：

1. OMDM 修订稿正文
2. 学生预习版讲义
3. 章节使用说明
4. 与本章相关的 SK04 / SK05 / SK06 规则

输出：

1. 本章原始内容清单
2. 本章主线节点表

### P2：重组教师版讲义

教师版讲义应优先包含：

1. 章节定位
2. 章节目标
3. 章节验收证据
4. 章节项目
5. 单元展开
6. 教师提示
7. 作业与收束

单元内部应同时体现：

1. `OMDM`
2. `Concept / Mechanism / Application`
3. 与 assignment pack 一致的证据字段
4. 与 memory 一致的高频误区提醒

### P3：生成学生预习版

学生预习版不是教师版删掉教师提示的缩写，而应：

1. 保留章节主线
2. 保留最小机制链
3. 弱化教师调度语言
4. 强化观察点、判断点、关系图要求和 AI 审核要求

### P4：质量复核

至少检查三层：

1. `外层外壳`
   是否保留稳定讲义结构
2. `内层主线`
   是否体现“场景 -> 数据流/边界 -> 机制 -> 观察/证据 -> 防护/收束”
3. `执行对齐`
   是否与 profile、assignment pack、评分规则、memory 一致

### P5：正式纳入资产目录

通过复核后，才将章节讲义挂入：

1. `50_assets/章节讲义/`
2. `50_assets/README_v5.0.md`
3. 根目录 `README.md`

## 6. 质量门槛

5.0 正式讲义至少应满足：

1. 不得与本章 profile 冲突
2. 不得与本章 assignment pack 的任务、红线、证据字段冲突
3. 必须显式或隐式体现 `OMDM`
4. 必须能支撑 `Concept / Mechanism / Application`
5. 必须体现 `证据 -> 结论` 约束
6. 必须把高频误区前置，而不是只在批改后再发现
7. 必须保留 AI 输出审核与人工复核的责任表达
8. 不得把 4.0 的旧字段直接搬进 5.0 而不做校验

## 7. 当前不做什么

当前阶段明确不做以下动作：

1. 不整体复制 `projects4.0/50_assets/章节讲义/`
2. 不一次性生成八章正式讲义
3. 不在缺少 profile / assignment pack / memory 的情况下强行补讲义
4. 不先做 docx 再回头补 markdown 源文件
5. 不把讲义迁移做成与 Step11、评分规则、教学反馈脱节的独立工程

## 8. Application

当前推荐的实际执行顺序是：

1. 先用第二章作为 `5.0 讲义迁移回归样本`
2. 再按第四到第八章依次迁移教师版讲义
3. 每迁一章，就补一章的 `4.0 -> 5.0讲义 -> 5.0作业包` 对照表
4. 等第二批章节也完成真实课堂回归后，再考虑补第一章和第三章

## 10. 反思问题

每次准备迁一章前，先问三个问题：

1. 这章现在的 assignment pack 和评分规则已经稳定了吗？
2. 这章讲义里哪些内容是 4.0 可以直接继承的，哪些必须按 5.0 重组？
3. 这章迁完后，能不能真正进入“讲义 -> 作业 -> 评分 -> feedback -> memory”的闭环？
