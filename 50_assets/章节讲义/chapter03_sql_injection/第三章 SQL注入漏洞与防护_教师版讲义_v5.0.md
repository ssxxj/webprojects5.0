# 第三章 SQL注入漏洞与防护 教师版讲义 v5.0

## 一、章节定位

第三章是整门课里第一个真正进入 `漏洞原理 -> 现象 -> 机制 -> 防护` 闭环的核心章节。

它的教学重点不是让学生记住若干 payload，而是让学生稳定理解三件事：

1. 用户输入为什么会影响数据库查询结构
2. 为什么看上去只是“输入一个值”，实际上可能改变系统执行语义
3. 为什么真正的防护不是限制几个字符，而是把数据与指令分离

本章如果讲成“字符串技巧课”，后面学生会出现两个系统性问题：

1. 把 SQL 注入理解成“特殊字符问题”
2. 把防护理解成“多过滤几个输入”

所以本章必须始终围绕这条主线推进：

`输入 -> 拼接 -> 解析 -> 执行 -> 回显/行为差异 -> 结构分离防护`

## 二、章节目标

1. 学生能解释 SQL 注入的形成条件与根因。
2. 学生能在 DVWA 中观察和复现基础 SQL 注入现象。
3. 学生能区分普通 SQL 注入与盲注在证据层的差异。
4. 学生能比较 Low 与 Medium 在控制思路上的差异。
5. 学生能从参数化查询、错误处理和最小权限角度解释防护链条。

## 三、章节验收证据

1. 能用自己的语言解释 SQL 注入为什么成立。
2. 能完成 DVWA SQL Injection Low 基础实验并说明现象。
3. 能比较 Low 与 Medium 在控制思路上的差异。
4. 能说明盲注为什么依赖行为差异而不是直接回显。
5. 能完成 `SQL注入因果链/防护关系图`。
6. 能完成 `AI输出审核与人工复核记录`。

## 四、章节主线

### Concept

SQL 注入不是“输入里有某些危险字符”，而是“不可信输入进入了数据库会解释执行的查询结构”。

### Mechanism

如果学生看不见“结构被污染”这一层，就会持续把注意力放在：

1. 某个 payload 长什么样
2. 某个页面回显了什么
3. 某个技巧能不能复现

而忽略真正应稳定迁移的分析框架：

`输入 -> 查询构造 -> 数据库解释 -> 结果差异 -> 根因控制`

### Application

教师授课时应反复要求学生回答三个问题：

1. 这里看到的是“正常查询结果变化”，还是“查询逻辑被改变”？
2. 这组证据支持的是“现象判断”，还是“根因判断”？
3. 当前防护措施控制的是根因、暴露面，还是后果范围？

## 八、单元展开

### 单元1 数据库与查询条件基础

#### 单元目标

1. 学生理解数据库在 Web 应用中的作用。
2. 学生能说出查询对象、查询条件与返回结果的最小关系。
3. 学生为理解 SQL 注入建立必要前置知识。

#### 达成证据

学生能用自然语言说明：

`输入条件 -> 数据库查询 -> 返回结果`

#### Observation

展示一个“输入 ID 查用户信息”或“输入用户名查记录”的业务场景，让学生先说表面上发生了什么。

#### Decision

让学生判断：

`页面变化只是浏览器显示不同，还是背后查询条件进入了数据库处理？`

#### Evidence

要求学生指出：

1. 哪个输入会作为查询条件
2. 为什么不同输入会得到不同结果
3. 哪些业务天然依赖数据库

#### Mechanism

教师收束：

SQL 注入首先是“系统根据用户输入构造查询”的问题，而不是数据库突然出错。

#### Modification

追问：

如果页面功能根本不查数据库，输入变化还会不会带来同类结果差异？

#### Concept

数据库、查询条件、返回结果是后续注入分析的最小结构图。

#### Mechanism

如果学生不理解“输入如何进入查询条件”，后面只会机械记录现象，无法解释根因。

#### Application

任务1 不要求学生掌握完整 SQL 语法，但要求他们能把页面功能与数据库查询关系说清楚。

#### 常见误区与纠偏

1. 误区：数据库和页面输入没什么关系。
   纠偏：登录、搜索、用户信息查询等场景都和数据库紧密相关。
2. 误区：不会写 SQL 就学不了 SQL 注入。
   纠偏：本章首先要求理解“输入如何影响查询”。

#### 形成性评估

要求学生用一句话回答：

`为什么说 SQL 注入首先是一个“系统根据输入构造查询”的问题？`

### 单元2 SQL注入 Low 现象观察

#### 单元目标

1. 学生能在 DVWA 中观察基础 SQL 注入现象。
2. 学生能描述正常输入与异常输入导致的差异。
3. 学生形成 SQL 注入的第一层直观认识。

#### 达成证据

学生能说清：

1. 输入发生了什么变化
2. 页面或结果发生了什么差异
3. 为什么这种差异不像普通查询变化

#### Observation

展示 DVWA SQL Injection Low 中同一输入框在不同输入下返回结果发生明显变化的现象。

#### Decision

让学生判断：

`这只是正常查询条件不同，还是查询逻辑本身被改变了？`

#### Evidence

要求学生比较：

1. 普通输入时的结果
2. 异常输入时的结果
3. 结构、数量、提示或表现上的差异

#### Mechanism

教师收束：

SQL 注入 Low 的教学价值，不是复现某个样例，而是帮助学生看见“输入变化已经不再只是查不同值”。

#### Modification

追问：

如果系统把输入只当成普通数据、不让它影响查询结构，还会不会出现同样差异？

#### Concept

现象观察是入口，不是结论本身。

#### Mechanism

只有把结果差异和“查询结构被影响”联系起来，现象才转化为机制证据。

#### Application

任务2 的最低要求不是“有截图”，而是截图能支撑差异比较。

#### 常见误区与纠偏

1. 误区：只要结果变了，就是漏洞。
   纠偏：关键要看变化是否说明查询逻辑异常。
2. 误区：先记 payload 比先理解现象更重要。
   纠偏：没有机制观察，payload 只会变成记忆碎片。

#### 形成性评估

让学生用一句话说明：

`为什么这里看起来不像正常查询变化？`

### 单元3 SQL注入 Low 机制分析

#### 单元目标

1. 学生能把 Low 级别现象推进到机制解释。
2. 学生能说出输入如何进入查询语句并影响结果。
3. 学生能画出最小因果链。

#### 达成证据

学生能说明：

`输入 -> 拼接 -> 解析 -> 执行 -> 结果差异`

#### Observation

回到上一单元已观察到的典型样例，让学生重新梳理请求参数、返回结果和页面现象。

#### Decision

让学生判断：

`问题核心在某个特殊字符本身，还是在输入进入了不该进入的查询结构？`

#### Evidence

要求学生在图中明确标出：

1. 输入点
2. 参数位置
3. 查询结构受影响的位置
4. 结果差异

#### Mechanism

教师收束：

Low 级别最重要的不是样例，而是让学生看见“拼接、解析、执行”三步。

#### Modification

追问：

如果改成参数化查询，这条链会在哪一步被打断？

#### Concept

SQL 注入的根因是结构被污染，不是某个字符天生危险。

#### Mechanism

把 payload 写成根因，会导致学生在后续所有注入类问题里都把注意力放错层级。

#### Application

任务3 必须要求学生写清“根因是结构被污染”，而不是只写“输入里有特殊字符”。

#### 常见误区与纠偏

1. 误区：只要记住某个 payload 就等于理解了 SQL 注入。
   纠偏：payload 只是入口，结构污染才是根因。
2. 误区：拼接和执行是一回事。
   纠偏：拼接是构造语句，执行是数据库按结构解释并运行。

#### 形成性评估

让学生口头说明：

`为什么 SQL 注入的核心不是某个字符，而是输入进入了不该进入的结构？`

### 单元4 Low 与 Medium 对照

#### 单元目标

1. 学生能比较 Low 与 Medium 的差异。
2. 学生理解简单过滤为何常常不足。
3. 学生开始从“现象难易”转向“控制层次”思考。

#### 达成证据

学生能说明：

1. Medium 相对 Low 改进了什么
2. 哪些改动控制的是输入表面，哪些控制的是根因

#### Observation

展示 Low 与 Medium 在相似输入条件下的差异。

#### Decision

让学生判断：

`禁止部分输入是否已经等于真正安全？`

#### Evidence

要求学生列出 Medium 相对 Low 的改动，并判断这些改动控制的是：

1. 输入表面
2. 回显表面
3. 结构根因

#### Mechanism

教师收束：

Medium 的教学价值不在“更难复现”，而在于让学生看到“有限过滤”和“结构性防护”的区别。

#### Modification

追问：

如果过滤规则再换一组字符，根因会消失吗？如果改成参数化查询，变化又是什么性质？

#### Concept

Low 与 Medium 对照是控制思路对照，不是难度排行榜。

#### Mechanism

只有看见“控制层级差异”，学生才会真正理解防护设计。

#### Application

任务3 的差异比较必须回到“控制思路”，不能只写“Medium 更复杂”。

#### 常见误区与纠偏

1. 误区：能挡住几个输入就说明防住了。
   纠偏：要看根因是否被切断。
2. 误区：Medium 的价值只是“更难做出来”。
   纠偏：更重要的是帮助学生理解控制思路层次。

#### 形成性评估

让学生说明：

`Medium 相对于 Low 最有价值的一项改动是什么，为什么仍然不够？`

### 单元5 Blind SQL Injection 基础认识

#### 单元目标

1. 学生理解盲注与普通 SQL 注入在证据层的差异。
2. 学生能观察行为差异、真假提示或长度差异。
3. 学生能把“无直接回显”纳入同一机制链。

#### 达成证据

学生能说明：

1. 普通注入依赖什么证据
2. 盲注依赖什么证据
3. 两者共享什么根因

#### Observation

展示页面不直接显示查询结果，但输入变化仍导致真假提示或行为差异变化的现象。

#### Decision

让学生判断：

`页面没有直接回显，是否就等于系统没有泄露信息？`

#### Evidence

要求学生列出盲注场景中可观察的证据类型：

1. 提示真假
2. 页面长度变化
3. 行为差异
4. 状态差异

#### Mechanism

教师收束：

盲注不是另一套完全不同的问题，而是同一根因在“证据层变弱”后的表现。

#### Modification

追问：

如果连真假提示都没有，是否还可能通过更间接的差异观察到信息？

#### Concept

盲注的关键不是“更神秘”，而是“证据更间接”。

#### Mechanism

学生一旦明白“根因相同、证据不同”，后续迁移到命令执行、XSS 等章节时就更容易形成跨章节理解。

#### Application

任务3 里应明确要求学生写出：

`普通注入与盲注共享同一根因，只是证据形式不同`

#### 常见误区与纠偏

1. 误区：盲注就是更复杂的 payload。
   纠偏：教学重点应放在证据形式变化上。
2. 误区：看不到结果就无法分析。
   纠偏：行为差异本身就是证据。

#### 形成性评估

让学生用一句话解释：

`盲注为什么叫“盲”？`

### 单元6 SQL注入防护链

#### 单元目标

1. 学生能从根因角度理解 SQL 注入防护。
2. 学生能区分根因控制、暴露面控制和后果范围控制。
3. 学生能形成一份结构化防护清单。

#### 达成证据

学生能把以下措施映射到不同控制层：

1. 参数化查询
2. 输入校验
3. 错误处理
4. 最小权限

#### Observation

回顾 Low、Medium 和盲注中看到的现象，让学生先说“要切断这些现象，优先卡在哪些位置”。

#### Decision

让学生判断：

`防护重点应放在猜学生会输入什么，还是不让输入进入 SQL 结构？`

#### Evidence

要求学生把每项防护措施映射到：

1. 根因
2. 暴露面
3. 后果范围

#### Mechanism

教师收束：

参数化查询控制根因，错误处理和回显控制暴露面，最小权限控制后果范围。

#### Modification

追问：

如果根因没解决，只做报错隐藏和权限收缩，风险会怎样变化？

#### Concept

防护不是建议列表，而是控制链设计。

#### Mechanism

只有把措施放回风险链，学生才不会把“更多建议”误当成“更强理解”。

#### Application

任务4 的作业必须检查：学生是否说明每条建议控制的是哪一层，而不是只列名词。

#### 常见误区与纠偏

1. 误区：把危险字符都拦掉就够了。
   纠偏：结构分离才是根因控制。
2. 误区：隐藏报错就说明安全了。
   纠偏：那只是降低暴露面，不等于根因消失。

#### 形成性评估

让学生说明：

`如果只能保留一条最关键的 SQL 注入防护原则，你会选什么，为什么？`

### 单元7 项目收束与迁移

#### 单元目标

1. 学生能把原理、现象、防护整合成结构化表达。
2. 学生能用证据支撑自己的判断。
3. 学生能把本章共性迁移到后续章节。

#### 达成证据

学生能提交并口头讲解一份 SQL 注入实验分析报告，说明：

1. 为什么成立
2. 看到了什么
3. 如何防护

#### Observation

回看学生自己的实验记录、差异对照表和因果图，让学生先说“我手里现在有哪些证据”。

#### Decision

让学生判断：

`报告最重要的是截图多，还是判断、证据和机制链讲清楚？`

#### Evidence

要求学生至少使用：

1. 一组现象截图
2. 一组差异记录
3. 一张机制链图

#### Mechanism

教师收束：

章节项目不是重复做实验，而是训练学生把“观察—判断—证据—机制—防护”收束成完整分析。

#### Modification

追问：

如果把本章根因逻辑迁移到命令执行或 XSS，你能保留什么共性？

#### Concept

项目报告是把实验操作转化为机制表达的工具。

#### Mechanism

只有当学生能把原理、现象、防护和证据放到同一条链上时，才说明真正理解了本章。

#### Application

收口项评分应强调：

1. 因果链是否完整
2. 防护关系是否对位
3. 是否存在结论超范围

#### 常见误区与纠偏

1. 误区：项目报告就是把步骤按时间顺序写一遍。
   纠偏：更重要的是围绕判断、证据和机制组织。
2. 误区：只要有截图，解释可以很短。
   纠偏：截图必须服务论证，而不是代替论证。

#### 形成性评估

让学生用一句话解释：

`如果只用一句话向别人解释 SQL 注入，你会怎么说？`

## 九、教师提示

1. 第三章不能讲成 payload 记忆课，要始终围绕“输入如何进入执行结构”推进。
2. Low 负责帮助学生看见根因，Medium 负责帮助学生看见“有限改动不等于根因消除”。
3. 盲注要强调证据形式变化，而不是技巧炫示。
4. 防护部分必须明确区分根因控制、暴露面控制和后果范围控制。
5. 若学生已出现“把 payload 写成根因”的表述，应立即用 `证据 -> 结论` 规则纠偏。

## 十、课后衔接

   继续沿着“输入进入不该进入的结构”理解上传与命令执行风险。
