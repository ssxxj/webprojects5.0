# 第二章 Web信息收集与请求分析 教师版讲义 v5.0

## 一、章节定位

第二章的功能不是“介绍几个工具”，而是把学生从第一章的系统模型，推进到“面对真实页面与请求时，知道先观察什么、为什么观察、哪些结果只能算线索、哪些结果才算证据”。

本章在整门课中的桥接作用是：

`系统工作原理 -> 观察对象与证据 -> 后续漏洞章节中的验证与防护`

因此，本章的真正产出不是“会用多少工具”，而是建立一条稳定的分析主线：

`公开线索 -> 第三方索引 -> 页面与请求观察 -> 参数差异 -> 授权主动探测 -> 风险与边界表达`

## 二、章节目标

1. 学生能区分公开线索、第三方索引线索、真实请求证据与主动探测结果。
2. 学生能在授权边界内完成第二章六类基础任务。
3. 学生能说明不同工具分别处理哪一层信息，以及为什么会得到这类结果。
4. 学生能把六层信息组织成关系链，而不是写成孤立工具清单。
5. 学生能对边界、脱敏、风险和下一步选择给出负责任表达。

## 三、章节验收证据

1. 能提交六项任务的最小证据与解释。
2. 能画出六层信息综合关系图/表，说明“能说明什么、不能替代什么、下一步为什么是这个”。
3. 能用一组请求差异说明参数变化如何支撑机制猜测。
4. 能在 Burp 与 Nmap 相关部分写清授权边界和风险表达。
5. 能完成 AI输出审核与人工复核记录，而不是直接复制工具输出。

## 四、章节主线

### Concept

本章主线是“从线索走到证据，再从证据走到可控验证”。

### Mechanism

如果学生不能分清结果层级，就会出现两类系统性错误：

1. 把 Google、Whois、Shodan 这类结果直接写成“已证明漏洞”。
2. 把 Burp/Nmap 的局部结果无限放大成“系统整体结论”。

### Application

教师授课时应持续用以下四层语言收束：

1. 这是什么层级的信息。
2. 它能说明什么。
3. 它不能替代什么。
4. 下一步为什么应是这个。

## 五、章节项目

- 项目名称：`DVWA 本地靶场六层信息观察与证据链整理`
- 项目交付物：
  1. 六类任务记录
  2. 六层信息综合关系图/表
  3. 风险 / 防护说明
  4. 学生自评表
  5. AI输出审核与人工复核记录

## 八、单元展开

### 单元1 观察对象、授权边界与六层主线

#### 单元目标

1. 学生理解第二章的真正目标是建立信息分层与证据意识。
2. 学生知道本章所有实践必须限定在授权环境内。
3. 学生能说出六层信息之间的基本关系。

#### 达成证据

学生能用自己的话说清：

1. 线索、证据、主动探测三者的区别。
2. 为什么第二章的顺序不应直接跳到扫描或漏洞判断。

#### 先修知识

1. 第一章中的 URL、请求、响应、状态、Cookie 基本概念。
2. 浏览器与服务器之间的基本交互模型。

#### Observation

给学生看同一目标可能出现的五类结果样例：

1. Google 搜索结果
2. Whois 域名字段
3. Shodan 端口记录
4. Burp 中的请求与响应
5. Nmap 的端口状态

先不解释，只要求学生描述“看见了什么类型的信息”。

#### Decision

判断题：

`这些结果中，哪些只能算线索，哪些更接近证据，哪些属于主动探测？`

#### Evidence

要求学生把五类结果按“页面 / 域名 / 服务 / 请求 / 主动探测”做第一轮归类，并说明归类依据。

#### Mechanism

教师收束：

1. Google 更接近公开页面线索。
2. Whois 更接近域名身份线索。
3. Shodan 更接近第三方历史索引线索。
4. Burp 更接近真实请求交互证据。
5. Nmap 属于授权条件下的主动探测结果。

#### Modification

追问：

1. 如果把 Shodan 写成“我已经探测到目标实时开放该端口”，哪里错了？
2. 如果把 Nmap 扫描结果写成“已经证实漏洞成立”，哪里又超范围了？

#### Concept

第二章的核心概念是“信息层次”。不同工具不是在同一层竞争，而是在不同层回答不同问题。

#### Mechanism

学生最容易失败的地方，不是不会用工具，而是不会控制结论范围。没有分层，就会把所有结果混成“已经发现问题”。

#### Application

本单元结束时，要求学生先写出本章关系图的骨架：

`上一层结果 -> 能说明什么 -> 不能替代什么 -> 下一步`

#### 常见误区与纠偏

1. 误区：会更多工具就说明分析更强。
   纠偏：真正重要的是知道工具对应哪一层信息。
2. 误区：课堂提到的工具都可以去公网练。
   纠偏：教学介绍不等于授权。

#### 形成性评估

让学生现场完成一张最小分类表，并口头解释其中一行为什么不能直接推出漏洞结论。

### 单元2 页面结构、开发者工具与 Burp 抓包

#### 单元目标

1. 学生能从页面结构定位输入点、输出点和状态对象。
2. 学生能在 Burp 中找到请求方法、路径、参数、Cookie 与响应位置。
3. 学生开始建立“页面 -> 请求 -> 响应”的观察链。

#### 达成证据

学生能指出一个 DVWA 页面中：

1. 至少 1 个输入点
2. 至少 1 个输出点
3. 至少 1 个状态对象
4. Burp 中与该页面动作对应的一组请求证据

#### Observation

先展示 DVWA 页面，再展示同一动作在 Burp 中出现的请求。

#### Decision

判断题：

`抓包是在重复浏览器已经显示的东西，还是把不可见的交互证据显性化？`

#### Evidence

要求学生在 Burp 截图上标出：

1. Method
2. URL/路径
3. 参数位置
4. Cookie 或 Header
5. 状态码或返回含义

#### Mechanism

教师收束：

页面只是表象，真正可分析的对象是请求与响应。Burp 的教学价值不在“功能多”，而在“把因果链拉出来”。

#### Modification

如果页面动作触发了多个请求，先看哪个？

收束标准：

优先看与页面功能目标直接相关、且能解释页面变化的那一个。

#### Concept

请求证据是“页面结果为何出现”的可追溯表达。

#### Mechanism

学生如果只看页面、不看请求，就会停留在现象层；一旦能定位请求对象，后续所有漏洞章节才有真正的证据入口。

#### Application

把作业任务 5 的最小字段当堂演示一遍，明确“缺字段不是格式问题，而是证据链断裂”。

#### 常见误区与纠偏

1. 误区：抓到请求就等于会分析。
   纠偏：必须继续看字段、差异和返回含义。
2. 误区：只要截图很多就说明证据充分。
   纠偏：关键是截图能否支撑结论。

#### 形成性评估

让学生用一句话回答：

`为什么 Burp 比 Google / Whois / Shodan 更接近技术证据？`

### 单元3 参数变化、差异观察与证据链

#### 单元目标

1. 学生理解差异观察的价值。
2. 学生知道“参数变化 -> 返回变化 -> 机制猜测”的最小分析链。
3. 学生能避免把差异直接夸大成漏洞结论。

#### 达成证据

学生能记录至少一组：

`原始参数 -> 修改参数 -> 响应差异 -> 暂时结论`

并说明为什么该结论仍是“受限结论”。

#### Observation

演示同一请求在 Repeater 中修改一个参数后，页面提示、状态码或返回长度出现变化。

#### Decision

判断题：

`看到差异后，能否直接说“已经证实漏洞存在”？`

#### Evidence

要求学生把差异写成结构化记录，而不是口头说“变了”。

#### Mechanism

教师收束：

差异说明“系统处理依赖了这个输入”，但不能自动推出“就是漏洞”。第二章只能走到“差异是机制线索”，不能越界到“漏洞成立”。

#### Modification

追问：

1. 如果只有返回长度变了，但页面看起来差不多，要不要记录？
2. 如果参数变化后几乎没变化，是否说明该参数完全不重要？

#### Concept

第二章强调的是“证据驱动的机制猜测”，不是“漏洞结论抢跑”。

#### Mechanism

`E2` 高频出现的根源，就是学生没有建立“证据层级”和“结论范围”的约束。

#### Application

此处要前置讲评：

1. 允许写：`该参数可能参与了系统处理`
2. 不允许直接写：`该参数已经证明存在漏洞`

#### 常见误区与纠偏

1. 误区：差异越大才值得记录。
   纠偏：长度、状态码、提示语的小变化同样可能重要。
2. 误区：有差异就等于漏洞。
   纠偏：差异只能说明系统对输入有依赖。

#### 形成性评估

给出一组差异记录，让学生判断哪一句结论超范围，并解释为什么。

### 单元4 公开线索、第三方索引与主动探测

#### 单元目标

1. 学生能把 Google、Whois、Shodan、Nmap 放回正确的信息层。
2. 学生能说清“能说明什么 / 不能替代什么”。
3. 学生能理解 Nmap 的授权边界。

#### 达成证据

学生能完成第二章任务 1、2、3、6 的最小字段，并写出至少一组边界说明。

#### Observation

展示四类结果样例：

1. 搜索结果页
2. 域名注册字段
3. 第三方端口与服务索引
4. Nmap 主动探测结果

#### Decision

判断题：

`如果目标是本地 DVWA 页面分析，最相关的观察层是哪一层？哪些层只用于建立分层意识和边界意识？`

#### Evidence

要求学生完成四行表格：

1. 工具名称
2. 处理哪层信息
3. 能说明什么
4. 不能替代什么

#### Mechanism

教师收束：

1. Google 帮你缩小公开页面范围。
2. Whois 帮你看到域名身份背景。
3. Shodan 帮你看到第三方索引中的服务暴露线索。
4. Nmap 是本人主动发送探测后得到的结果，因此边界最敏感。

#### Modification

追问：

1. 如果作业是在 DVWA 本地环境完成，为什么仍然要介绍 Google / Whois / Shodan？
2. 如果场景换成未授权真实系统，Nmap 为什么不能随便做？

#### Concept

不同工具是“分层观察器”，不是“统一答案机”。

#### Mechanism

学生若把这些层混在一起，就会出现两个后果：

1. 线索写成证据
2. 认识工具变成越界实践

#### Application

本单元应明确：

1. Google / Whois / Shodan 重在“定位 + 层级 + 边界”
2. Nmap 重在“授权主动探测”的最小认识

#### 常见误区与纠偏

1. 误区：Shodan 的结果就是我当前实时看到的结果。
   纠偏：Shodan 是第三方历史索引。
2. 误区：开放端口就说明漏洞成立。
   纠偏：开放端口只说明攻击面，不自动说明漏洞。

#### 形成性评估

让学生用一句话分别完成：

1. `Shodan 能说明什么`
2. `Shodan 不能替代什么`
3. `Nmap 为什么必须建立在授权前提下`

### 单元5 社工风险、关系图与责任表达

#### 单元目标

1. 学生能用社工案例说明“技术观察之外的人与组织风险”。
2. 学生能完成六层信息综合关系图/表。
3. 学生能完成自评和 AI 输出审核记录。

#### 达成证据

1. 至少 1 个社工案例分析
2. 1 张六层信息综合关系图/表
3. 1 份学生自评表
4. 1 份 AI输出审核与人工复核记录

#### Observation

先给一个社工案例，再给几份典型关系图样例：

1. 只列工具名的错误样例
2. 写出“能说明什么 / 不能替代什么 / 下一步”的合格样例

#### Decision

判断题：

`综合关系图到底是在总结工具，还是在总结“如何从上一层走到下一层”的决策链？`

#### Evidence

要求学生把关系图至少写出以下列：

1. 信息层
2. 方法/对象
3. 结果类型
4. 能说明什么
5. 不能替代什么
6. 下一步

#### Mechanism

教师收束：

1. `R2` 的本质不是“没画图”，而是“不会表达关系”。
2. 本章最后的收束项，不是把工具再抄一遍，而是证明自己已经知道“下一步为什么是这个”。

#### Modification

追问：

1. 如果一张图里把 Google、Whois、Shodan、Burp、Nmap 全列了，但没有下一步和边界说明，为什么不能给满分？
2. 如果学生使用了 AI 生成文字，但没有审核，风险在哪里？

#### Concept

综合关系图评估的是结构化理解，自评与 AI 审核评估的是责任意识与元认知。

#### Mechanism

项目课不是只看“做出来没有”，还要看“是否知道自己在做什么、结论边界在哪里、哪些话是自己核实过的”。

#### Application

本单元务必把以下要求说透：

1. 图已提交但只列工具，不等于关系达标。
2. 自评不是打勾，而是说明自己最不确定什么。
3. AI 审核不是交代“用了没”，而是交代“保留了什么、删改了什么、核实了什么”。

#### 常见误区与纠偏

1. 误区：关系图只是收尾形式。
   纠偏：关系图是本章主线是否形成的最直接证据。
2. 误区：AI 审核就是形式表态。
   纠偏：它检查的是责任意识和自我校验能力。

#### 形成性评估

让学生现场指出一张错误关系图的问题，并改写其中一行“下一步”。

## 九、教师提示

1. 第二章不是“搜集更多信息”，而是“限制结论范围”。
2. 每次讲工具，都要同步讲“能说明什么 / 不能替代什么”。
3. 每次讲证据，都要同步讲“证据支撑到哪里就停在哪里”。
4. 每次讲 Burp / Nmap，都要同步讲授权边界。
5. 高频误区 `R2 / E1 / E2 / T2` 必须在课堂前置，而不是等作业提交后再批评。

## 十、课后衔接

1. 与作业包直接对齐，学生完成六个任务和关系图。
2. 与第三章衔接时，明确告诉学生：
   - 第三章不是突然开始“打漏洞”
   - 而是在第二章证据链之上进入“输入如何影响查询结构”的机制分析

## 十一、反思问题

1. 如果学生把线索和证据分不清，后面哪几章会一起出问题？
2. 如果学生会抓包，但不会控制结论范围，这说明他缺的是操作还是机制？
3. 如果学生画了图但没有关系，他真正缺的是格式、概念，还是决策链？
