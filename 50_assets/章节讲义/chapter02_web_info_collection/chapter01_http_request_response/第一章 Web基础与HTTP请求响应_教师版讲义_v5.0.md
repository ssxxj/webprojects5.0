# 第一章 Web基础与HTTP请求响应 教师版讲义 v5.0

## 一、章节定位

第一章不是漏洞章节，而是整门课的认知起点。

它真正要搭的不是某个技术点，而是三层基础脚手架：

1. `系统脚手架`
   浏览器、服务器、数据库不是孤立名词，而是一条数据流链。
2. `通信脚手架`
   页面变化背后对应请求、响应、参数、状态与返回结果。
3. `分析脚手架`
   后续所有章节最终都要回到：
   `输入 -> 处理 -> 输出 -> 执行 -> 身份`

如果这一章只讲成“概念导论”，后面学生就会出现典型问题：

1. 只记页面现象，不知道现象属于哪一层。
2. 只会说 HTTP、Cookie、Session 名词，不会解释它们为什么存在。
3. 后续遇到 SQL 注入、XSS、会话安全时，无法把漏洞放回系统链路理解。

## 二、章节目标

1. 学生能说出浏览器、服务器、数据库在 Web 应用中的基本角色。
2. 学生能解释一次 HTTP 请求与响应的大致流程。
3. 学生能识别 URL 参数、表单参数、Cookie、Session 等基础对象。
4. 学生能解释系统为什么需要状态维持，以及 Cookie 与 Session 的职责分工。
5. 学生能用“输入—处理—输出—执行—身份”框架分析简单页面场景。

## 三、章节验收证据

1. 能画出 Web 三层架构与简化数据流图。
2. 能对一个基础请求与响应样例指出参数、方法、返回结果和状态对象。
3. 能解释“为什么登录一次后页面还能继续识别你”。
4. 能把简单页面场景放回“输入—处理—输出—执行—身份”框架。
5. 能完成 `系统链路图/关系图`。
6. 能提交 `AI输出审核与人工复核记录`。

## 四、章节主线

### Concept

第一章的核心不是“认识几个术语”，而是把网页看成系统，把页面看成链路结果。

### Mechanism

学生之所以会在后面章节学碎，通常不是因为不会操作，而是因为最初没有建立这条稳定主线：

`页面现象 -> 请求与响应 -> 状态维持 -> 统一分析框架`

如果主线缺失，后面的漏洞知识就会退化成：

1. 一个页面一个技巧
2. 一个工具一个截图
3. 一个漏洞一个现象

### Application

教师授课时应持续追问三个问题：

1. 这个页面现象背后是哪条系统链路在工作？
2. 学生说出的术语，能不能回到证据和机制解释？
3. 当前内容如何成为后续漏洞章节的共同前置模型？

## 八、单元展开

### 单元1 课程导入、边界与伦理

#### 单元目标

1. 学生能区分授权实验与未授权操作。
2. 学生理解本课程不是工具炫技课，而是风险与防护机制课。
3. 学生建立“证据化表达边界”的基本责任意识。

#### 达成证据

学生能说明：

1. 某个实验场景为什么属于授权范围。
2. 为什么相同工具在不同对象上性质不同。

#### Observation

展示三个场景：

1. 在本机 DVWA 中观察登录页
2. 对陌生站点做主动扫描
3. 在教师授权靶场里做课堂任务

#### Decision

让学生先判断：

`判断合法性的核心，是“用了什么工具”，还是“有没有明确授权、范围和责任”？`

#### Evidence

要求学生指出判断依据：

1. 实验对象是谁
2. 授权从哪里来
3. 范围是否明确
4. 是否存在课堂要求和记录

#### Mechanism

教师收束：

技术动作本身不决定合法性，`边界、授权、范围、责任` 才决定性质。

#### Modification

追问：

如果把同样动作从 DVWA 换到未授权公网对象，性质会怎样变化？

#### Concept

课程第一单元先建边界模型，而不是先堆技术名词。

#### Mechanism

这是为了防止学生形成“安全课=攻击课”的错误脚本。

#### Application

后续每一章都应要求学生在证据截图、作业说明和 AI 审核中交代边界。

#### 常见误区与纠偏

1. 误区：只要不造成破坏，尝试一下没关系。
   纠偏：未授权本身已经越界。
2. 误区：课堂上学工具，就是以后可以随便用。
   纠偏：工具能力和使用边界必须同时学习。

#### 形成性评估

给出三个场景，要求学生判断是否属于授权实验，并写出至少两条依据。

### 单元2 Web三层架构与数据流

#### 单元目标

1. 学生能画出浏览器、服务器、数据库三层关系。
2. 学生能说明数据如何在三层之间流动。
3. 学生能把页面变化解释成系统处理结果。

#### 达成证据

学生能提交一张 `浏览器 -> 服务器 -> 数据库 -> 结果返回` 的简化图，并标出输入进入位置。

#### Observation

展示 DVWA 登录页、提交动作和登录后页面变化。

#### Decision

让学生判断：

`页面变化主要是浏览器自己完成，还是背后存在服务器处理与数据返回？`

#### Evidence

要求学生指出：

1. 输入点
2. 提交动作
3. 返回结果
4. 可能的后端处理位置

#### Mechanism

教师收束：

Web 页面是系统链路的结果，不是系统本身。

#### Modification

追问：

如果没有服务器处理，页面变化会是什么性质；如果没有数据库，这一页还能否呈现对应结果。

#### Concept

三层架构是后续所有安全分析的最小系统图。

#### Mechanism

后续漏洞都发生在这条链路的某个节点，而不是停留在页面表面。

#### Application

任务1 评分不能只看图画得像不像，而要看是否表达了：

1. 输入从哪进入
2. 处理在哪发生
3. 结果如何返回

#### 常见误区与纠偏

1. 误区：浏览器里看到的页面就是全部系统。
   纠偏：浏览器看到的是结果，不是全过程。
2. 误区：数据库离页面很远，所以安全关系不大。
   纠偏：很多风险正是沿着后端处理和数据读写形成的。

#### 形成性评估

要求学生口头解释：

`如果只盯着页面界面，为什么很难真正理解漏洞？`

### 单元3 HTTP/HTTPS请求与响应

#### 单元目标

1. 学生理解请求和响应是浏览器与服务器交互的最小载体。
2. 学生能识别方法、参数、返回结果与状态对象。
3. 学生能区分“传输保护”和“业务安全”不是同一层问题。

#### 达成证据

学生能对一个基础请求样例指出：

1. 请求方法
2. 参数位置
3. 返回结果
4. 状态信息

#### Observation

展示一个最小登录请求或查询请求的页面过程。

#### Decision

让学生判断：

`HTTPS 能解决的是“传输过程保护”，还是“应用逻辑一定安全”？`

#### Evidence

要求学生指出请求里的：

1. URL/路径
2. 参数
3. 状态码或页面结果
4. Cookie/状态线索

#### Mechanism

教师收束：

HTTP/HTTPS 是承载 Web 行为的通道，但 HTTPS 不能替代输入校验、输出控制和身份校验。

#### Modification

追问：

如果同样页面用了 HTTPS，为什么后续仍然可能存在 SQL 注入或 XSS？

#### Concept

请求与响应是观察 Web 安全问题的最小观察窗口。

#### Mechanism

后续章节里所有“输入影响系统”的问题，都要通过请求与响应被看见。

#### Application

任务2 不能只让学生写“HTTP 是请求协议”，而要要求他们把术语落到具体页面样例。

#### 常见误区与纠偏

1. 误区：只要是 HTTPS 网站就一定安全。
   纠偏：HTTPS 保护的是传输过程，不直接修复业务逻辑。
2. 误区：参数只是地址栏里的内容。
   纠偏：表单、请求体、Cookie 都可能携带输入。

#### 形成性评估

给出一个请求示意，要求学生指出至少一个“传输安全不等于业务安全”的例子。

### 单元4 Cookie与Session状态维持

#### 单元目标

1. 学生理解 HTTP 无状态这一前提。
2. 学生能区分 Cookie 与 Session 的职责。
3. 学生能用证据解释“系统为什么记得你”。

#### 达成证据

学生能说明：

1. 为什么登录一次后页面还能保持已登录状态。
2. Cookie 在浏览器侧做什么，Session 在服务器侧做什么。

#### Observation

展示登录前后页面状态差异，以及跳转到其他页面后状态仍保留的现象。

#### Decision

让学生判断：

`系统识别用户主要依赖 IP，还是依赖浏览器状态标识与服务器会话记录的配合？`

#### Evidence

要求学生指出：

1. 登录前后页面状态变化
2. Cookie 线索
3. 后续请求仍能识别用户的现象

#### Mechanism

教师收束：

HTTP 本身不会“自然记住你”，状态维持是浏览器侧与服务器侧协作出来的。

#### Modification

追问：

如果浏览器不保存 Cookie、换一个浏览器、或者状态标识被篡改，会发生什么。

#### Concept

Cookie 与 Session 不是两个同义词，而是状态维持系统里的两个不同部件。

#### Mechanism

如果学生只背术语而不理解职责分工，后面会把弱会话、CSRF、身份校验全部学混。

#### Application

任务3 评分要特别检查两点：

1. 是否真的区分了 Cookie 与 Session
2. 是否给出了支持判断的现象或证据

#### 常见误区与纠偏

1. 误区：登录成功后系统自然会记住用户。
   纠偏：那是状态维持机制在工作，不是自然现象。
2. 误区：Cookie 和 Session 完全一样。
   纠偏：一个偏浏览器侧，一个偏服务器侧，职责不同。

#### 形成性评估

要求学生用一句话分别说明浏览器、Cookie、Session 在状态维持中的作用。

### 单元5 统一分析框架与课程收束

#### 单元目标

1. 学生理解后续章节可以放回同一分析链。
2. 学生能用“输入—处理—输出—执行—身份”描述一个简单页面场景。
3. 学生为后续漏洞章节建立统一结构。

#### 达成证据

学生能把一个简单页面场景拆成：

1. 输入
2. 处理
3. 输出
4. 执行
5. 身份/状态

#### Observation

给出一个基础页面场景，让学生先找：

1. 输入点
2. 页面变化
3. 状态对象

#### Decision

让学生判断：

`如果只允许先标一个最关键分析节点，应该标输入、处理、输出、执行还是身份？`

#### Evidence

要求学生把页面中的对象放进统一链路图，而不是直接说漏洞名。

#### Mechanism

教师收束：

OWASP 类风险清单是地图，但课程真正训练的是把不同风险放回同一链路。

#### Modification

追问：

如果在输入加校验、在输出加编码、在执行前做权限检查，风险链会在哪一步被截断。

#### Concept

统一框架的作用，是把后续章节从“分散漏洞”收束成“同一系统的不同失控点”。

#### Mechanism

这一步决定学生后面是学“目录式知识点”，还是学“可迁移的分析框架”。

#### Application

任务4 与收口项都要强调：

1. 关系图不是名词清单
2. 关系图必须体现上一层结果为什么导向下一步

#### 常见误区与纠偏

1. 误区：OWASP Top 10 就是一张要背的列表。
   纠偏：它更像风险地图，真正稳定的是分析框架。
2. 误区：每个漏洞都是独立知识点。
   纠偏：后续大多数章节都能回到同一条链路理解。

#### 形成性评估

给学生一个简单页面场景，要求他们用统一框架指出至少两个风险点和一个控制点。

## 九、教师提示

1. 第一章最怕讲成“定义堆叠”，所以要一直从系统图和数据流出发。
2. 优先让学生说“为什么”，再讲术语名词。
3. 所有示例都要强调：页面现象只是入口，不能替代机制解释。
4. 第一章不是炫技章节，实验重点是观察、描述、解释和建模。
5. 若学生已经出现“截图很多但不会解释”，应立刻前置 `证据 -> 结论` 约束。

## 十、课后衔接

1. 与第二章衔接：
   从“系统链路与请求状态”过渡到“公开线索、请求证据与主动探测”。
2. 与第三章衔接：
   从“输入进入系统”过渡到“输入如何影响查询结构”。
3. 与第七章衔接：
   Cookie/Session 的基础理解会直接成为身份与会话安全前置知识。
