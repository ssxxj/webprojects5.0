# 第一章 Web基础与HTTP请求响应 学生预习版讲义 v5.0

## 1. 本章先学什么

第一章不是先学漏洞，而是先学：

1. 网页为什么是系统结果
2. 请求与响应在系统里扮演什么角色
3. 系统为什么需要状态维持
4. 后续所有章节为什么都能放回同一条分析主线

本章主线只有一句话：

`从页面现象走到系统链路，再走到统一分析框架`

## 2. 为什么第一章不能学成“概念背诵”

如果你只记住：

1. 浏览器
2. 服务器
3. 数据库
4. HTTP
5. Cookie
6. Session

但不会回答“它们为什么存在、彼此如何协作”，那么后面学漏洞时就会变成：

1. 只会看页面变化
2. 不会解释请求和状态
3. 不知道风险到底发生在哪一层

## 3. 本章真正要分清的四件事

### Concept

第一章最重要的是建立系统模型，而不是术语清单。

### Mechanism

如果系统模型不稳，后面的漏洞知识就会全部碎片化。

### Application

请先分清这四件事：

1. `页面`
   你看到的结果，不等于全部系统。
2. `请求与响应`
   页面变化背后对应浏览器与服务器的交互。
3. `状态维持`
   系统能继续识别你，是机制协作，不是自然记忆。
4. `统一分析框架`
   后面章节要回到：
   `输入 -> 处理 -> 输出 -> 执行 -> 身份`

## 4. 本章最小观察框架

打开一个页面时，先问自己五个问题：

1. 输入从哪里进入
2. 系统可能在哪里处理
3. 结果从哪里回到页面
4. 系统靠什么维持状态
5. 这个页面现象能放回哪条系统链路

## 5. 本章作业会要求你完成什么

你需要完成四类任务：

1. Web三层架构与数据流图
2. HTTP请求与响应基础记录
3. Cookie与Session状态维持机制说明
4. 输入—处理—输出—执行—身份框架应用

还要完成三类收口项：

1. 系统链路图/关系图
2. 学生自评表
3. AI输出审核与人工复核记录

## 6. 提交前最容易出错的地方

### 误区1：图画了，但不是系统链路

只写：

1. 浏览器
2. 服务器
3. 数据库

还不够。

至少还要写出：

1. 输入从哪进入
2. 处理在哪里发生
3. 结果如何返回
4. 下一步为什么会走向后续章节

### 误区2：写了 HTTP/HTTPS，但没说清边界

要避免把：

`HTTPS = 网站一定安全`

写成结论。

更稳的表达是：

`HTTPS 主要解决传输过程保护，不直接等于业务逻辑安全。`

### 误区3：把 Cookie 和 Session 当成一个东西

最低要求不是背定义，而是能说明：

1. 浏览器侧保存了什么
2. 服务器侧记录了什么
3. 为什么二者要配合

### 误区4：结论超出证据

例如：

1. 看到登录状态保持，就写“系统一定很安全”
2. 看到 Cookie，就写“完全靠 Cookie 识别用户”

这些都不够稳。

## 7. 关系图/关系表应该怎么做

第一章的关系图/表至少应体现：

1. 页面现象
2. 请求/响应
3. 状态维持
4. 统一分析框架
5. 后续课程主线

最低合格标准不是“列名词”，而是“写出关系”。

## 8. AI输出审核与人工复核记录怎么写

不要只写“用了 AI”或“没用 AI”。

你要写清：

1. 保留了什么内容，为什么保留
2. 删改了什么内容，为什么删改
3. 自己核实了哪几个专业判断
4. 哪个边界最容易误判

本项检查的不是“是否使用工具”，而是“是否会审核工具输出”。

## 9. 预习时先思考三个问题

1. 为什么说网页不是孤立页面，而是系统结果？
2. 为什么登录状态不是服务器“自然记住你”？
3. 为什么后面的漏洞最终都能回到“输入—处理—输出—执行—身份”这条主线？

## 10. 预习后你至少应能说出

1. 浏览器、服务器、数据库分别做什么
2. 请求和响应为什么是 Web 安全观察入口
3. Cookie 与 Session 的职责为什么不同
4. HTTPS 为什么不等于应用本身一定安全
5. 第一章为什么是后续所有章节的认知脚手架
