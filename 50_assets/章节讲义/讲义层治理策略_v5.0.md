# 讲义层治理策略 v5.0

## 1. 定位

本文件定义 `projects5.0` 中 `50_assets/章节讲义/` 的最终治理边界。

目标不是把讲义正文也强行压进章节主配置，而是在保持讲义教学表达弹性的前提下，防止它与 `profile / assignment pack / 课堂实施方案` 长期漂移。

## 2. 采用方案

当前采用 `方案 B`：

`讲义独立维护，但纳入 preflight，做轻量一致性检查，不并入 single source。`

## 3. Concept

讲义层和作业包层承担的系统功能不同：

1. `profile / assignment pack / lesson script / 课堂实施方案`
   更接近结构化约束与执行接口，适合纳入单一来源。
2. `章节讲义`
   更接近教师表达、机制展开与课堂叙事，不适合被完全压平为结构化字段。

因此，讲义层不应被当作“另一份 assignment pack”，但也不能完全脱离系统治理。

## 4. Mechanism

### 4.1 讲义层的治理边界

讲义层当前保持独立维护，但必须和章节主配置保持以下最低一致性：

1. 章节目录必须与 `chapter_id` 对应。
2. 教师版讲义与学生预习版讲义的文件名必须与 `chapter_title` 对应。
3. 每章必须具备：
   - 教师版讲义
   - 学生预习版讲义
   - 讲义迁移清单
   - `4.0讲义 -> 5.0讲义 -> 5.0作业包` 对照表
4. 讲义迁移清单中的：
   - `chapter_id`
   - `chapter_name`
   - `upstream_5_0.profile`
   - `upstream_5_0.assignment_pack`
   - `generated_assets`
   必须有效且可追溯。

### 4.2 讲义层不纳入单一来源的部分

以下内容允许教师基于课堂情况自主改写，不纳入 drift check：

1. 单元展开顺序的教学表达
2. 举例方式、问题链和类比方式
3. 板书语言与提示语
4. 面向某个班级的临场补充说明

### 4.3 preflight 的角色

preflight 对讲义层只做 `轻量一致性检查`，不做 `正文级字面漂移检查`。

也就是说：

1. 它检查“讲义是否还在系统里、标题是否对、清单是否可追溯”。
2. 它不检查“讲义正文是否逐字等于某份主配置生成结果”。

## 5. Application

### 5.1 哪些情况下必须回改讲义

出现以下变化时，必须回改讲义：

1. 章节标题变更
2. 章节主线发生方向性变化
3. assignment pack 的任务结构发生实质变化
4. 评分规则新增了必须前置到讲义层的高频误区或责任表达
5. 真实课堂反馈表明讲义主线与作业/评分明显脱节

### 5.2 哪些情况下不必回改讲义

仅出现以下变化时，通常不必立即修改讲义：

1. 评分脚本内部实现细节调整
2. lesson script 的节奏微调
3. preflight / CI / drift check 的工程入口变化
4. assignment pack 的非方向性措辞优化

### 5.3 维护责任

建议采用以下责任分工：

1. `30_runtime/chapter_master_configs/`
   负责结构化单一来源
2. `50_assets/章节讲义/`
   负责教学表达与机制展开
3. `preflight_projects5_v5.py`
   负责讲义层轻量一致性门禁
4. `SK08 + build_teaching_feedback_v5.py`
   负责真实课堂后的讲义回归触发判断

## 6. 当前结论

`projects5.0` 现阶段不把讲义层并入章节主配置，也不对讲义正文做严格 drift check。  
但讲义层已经正式进入发布前门禁，不再属于完全游离资产。
