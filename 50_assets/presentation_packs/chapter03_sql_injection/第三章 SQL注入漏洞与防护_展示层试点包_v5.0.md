## 3. 课堂演示建议

### 演示目标

让学生看见：

```text
输入不是单独产生风险；
输入进入查询结构后，数据库解释方式发生变化，风险才成立。
```

### 演示形态

第一版只做教师课堂口头 + 板书演示，不新增网页 demo。

推荐流程：

1. 写出普通查询：`SELECT ... WHERE id = 输入`
2. 让学生判断输入只是“值”，还是可能变成“结构的一部分”
3. 对比普通输入和异常输入的结果差异
4. 追问：这个差异最多能支持什么结论，不能直接支持什么结论
5. 改成参数化查询，再问这条链在哪一步被截断

### 不做什么

1. 不收集 payload 清单。
2. 不展示公网真实站点。
3. 不把“成功弹出/返回异常结果”当成唯一教学目标。

## 4. 学生导读

本章你要带走的不是几个 SQL 注入字符串，而是一条判断链：

```text
输入 -> 查询构造 -> 数据库解释 -> 结果差异 -> 根因判断 -> 防护分层
```

提交作业时，每个结论都要回到证据：

1. 我看到了什么现象？
2. 这个现象说明查询结构可能发生了什么变化？
3. 这个证据最多能支持到什么结论？
4. 这个证据还不能直接证明什么？
5. 哪个防护措施切断的是根因，哪个只是降低暴露或后果？

## 5. 作业发布摘要

发布给学生时保留这 5 条：

1. 文件命名：`<学号><姓名>SQL注入漏洞与防护.pdf`
2. 实验范围：只限 DVWA、本机、localhost、127.0.0.1 或教师授权环境。
3. 必交：四类任务、SQL注入因果链/防护关系图、学生自评、AI 输出审核与人工复核记录。
4. 关系图必须体现：输入、查询构造、数据库解释、结果差异、防护控制点。
5. 不要只贴 payload 和截图；必须说明证据如何支持结论。

## 6. 点评提示

### 班级级点评

如果多数学生只记 payload：

```text
下次先不增加 payload 数量，先回到“输入如何进入查询结构”这条链。
```

如果多数学生把 Blind 写成另一种根因：

```text
下次用“同一根因，不同证据形式”做一次 5 分钟对比。
```

如果多数学生混淆防护层次：

```text
下次把参数化、错误处理、最小权限分别放到“根因 / 暴露面 / 后果范围”三列里重排。
```

### 个人级点评句式

1. 你的截图不少，但结论跳得快；下一版先补“这个现象最多能说明什么”。
2. 你已经看到结果差异了，下一步要把差异放回“输入 -> 查询构造 -> 数据库解释”链条。
3. 你能说出参数化查询，但还要说明它切断的是根因，而不是只把错误隐藏起来。
4. 你的 AI 审核记录不能只写“已检查”，要写清你删掉或改写了哪两个判断。
