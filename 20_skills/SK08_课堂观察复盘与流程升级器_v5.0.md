# SK08 课堂观察复盘与流程升级器 v5.0

## Purpose

把课堂观察、学生作业结果、评分 debug 与教师判断整合成：

1. 课后复盘
2. 班级级教学反馈
3. 章节级改进行动
4. 可沉淀的流程升级项

与 4.0 相比，5.0 的重点不是“课后写个总结”，而是把复盘变成：

`观察 -> 诊断 -> 反馈 -> 模板升级 -> 下轮教学调整`

## Input Contract

至少应具备以下输入中的 3 类：

1. 课堂观察记录
2. 作业评分结果
3. 班级 debug JSON
4. 章节 profile
5. 教师自我判断
6. 当前问题台账

若 `10_memory` 已存在相近章节或相近任务的沉淀，复盘前还应优先查看：

1. `10_memory/诊断标签索引_v5.0.md`
2. `10_memory/误区库_v5.0.md`
3. `10_memory/案例库_v5.0.md`
4. `10_memory/班级教学反馈索引_v5.0.md`

若输入不足，必须明确“本次仅为最小复盘”，并标出下次需要补哪些数据。

## Process

1. 先对照 `10_memory`，判断这次问题是新问题还是历史重复问题。
2. 先判断目标达成度，而不是先归因学生是否认真。
3. 区分问题来源：
   - 讲义设计问题
   - 作业设计问题
   - 评分口径问题
   - 前置知识问题
   - 环境与组织问题
   - 安全/合规问题
4. 把高频问题转成诊断标签，而不是只写长评语。
5. 把问题分成两类：
   - 需要下次课补讲的内容
   - 需要升级模板、skill、runtime 或资产的内容
6. 若已有批量评分 debug，优先使用：
   - `40_evaluation/runtime/build_teaching_feedback_v5.py`
7. 若章节 profile 已稳定，优先把改动反馈到：
   - `40_evaluation/runtime/chapter_profiles/*.json`
   - `50_assets/assignment_packs/<chapter_dir>/`
8. 对红线问题、雷同问题和合规问题，只输出“待教师确认”的复核结论，不替代教师做纪律裁决。
9. 复盘结束后，把值得复用的内容回写到 `10_memory`，避免同类问题下一轮重复发生。

## Output Contract

默认输出至少包括：

1. 班级总体情况
2. 高频错误标签
3. 高风险问题统计
4. 可作为正例的样本
5. 建议下轮教学动作
6. 建议升级的系统文件
7. 哪些内容进入案例库、误区库或模板升级清单
8. 哪些内容应回写到 `10_memory`

运行落点：

1. 班级级反馈模板：`40_evaluation/班级级教学反馈报告模板_v5.0.md`
2. 自动生成脚本：`40_evaluation/runtime/build_teaching_feedback_v5.py`
3. 章节变量层：`40_evaluation/runtime/chapter_profiles/*.json`
4. 正式资产层：`50_assets/assignment_packs/`
5. 经验沉淀层：`10_memory/`
6. 迁移与成长层：`90_growth/`

## Automation Entry

当批量评分已经生成 debug JSON 后，可直接运行：

```bash
python3 40_evaluation/runtime/build_teaching_feedback_v5.py \
  --debug <debug.json> \
  --profile 40_evaluation/runtime/chapter_profiles/<chapter_profile>.json \
  --class-name <班级名>
```

默认会产出：

1. 教学反馈 Markdown
2. 教学反馈 PDF
3. 教学反馈摘要 JSON

若已产生一个或多个摘要 JSON，复盘后应继续运行：

```bash
python3 10_memory/runtime/memory_update_v5.py \
  --inputs <教学反馈摘要1.json> <教学反馈摘要2.json>
```

用于把班级反馈自动回写到 `10_memory`。

## Quality Gate

合格的 SK08 结果至少满足：

1. 至少形成 3 条可执行改进行动
2. 至少指出 1 个需要升级的系统文件
3. 不把所有问题都归因给学生
4. 不把所有问题都归因给 AI
5. 能区分“讲什么”和“改什么”
6. 能明确哪些问题进入下轮课堂，哪些进入资产升级
7. 红线问题保持教师最终确认边界
8. 至少回写 1 条可复用经验到 `10_memory`

## Failure Handling

若只有课堂印象、没有结构化数据：

1. 先形成最小复盘
2. 标出缺失的观察项和 debug 输入
3. 下次必须补齐：
   - 班级 debug JSON
   - 至少 1 份正例
   - 至少 1 份待补救样本
   - 至少 3 个高频标签
