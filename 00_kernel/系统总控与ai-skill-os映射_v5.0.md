# 系统总控与 ai-skill-os 映射 v5.0

## 1. 系统定位

### Concept
`projects5.0` 的目标不是继续堆文档，而是把《Web应用安全与防护》课程变成一套
可运行、可迁移、可诊断、可反馈的教学操作系统。

### Mechanism
相较于 `projects4.0`，5.0 有两个核心升级：

1. `Step 11 升级`
   从“生成作业包”升级为“生成可教学反馈的作业系统”。
2. `Evaluation 升级`
   从“评分规则”升级为“成绩层 + 诊断层 + 班级反馈层”的三输出系统。

系统层仍保持六层结构：

| 层 | 作用 | 对应目录 |
| --- | --- | --- |
| Kernel | 系统定位、母规则、质量门 | `00_kernel` |
| Memory | 章节变量、误区库、样例沉淀 | `10_memory` |
| Skills | 12 步流程主技能 | `20_skills` |
| Runtime | 运行手册、项目指令词、最小清单 | `30_runtime` |
| Evaluation | 模板、Rubric、脚本、反馈 | `40_evaluation` |
| Growth | 教师成长、迁移、复盘 | `90_growth` |

### Application
5.0 的默认运行方式是：

1. 先判断当前处于 12 步中的哪一步
2. 再由 skill 路由到对应产出
3. 输出不仅要“可用”，还要“可评、可诊断、可复盘”
4. 将班级结果沉淀回 memory 和 growth

## 3. v5.0 的 Step 11 定义

Step 11 不再只输出：
- 任务书
- 学生版说明单
- 提交模板
- 自评表
- 教师验收表

还必须同步输出：
- 目标对齐表
- 任务分值拆分表
- 红线与维度扣分表
- AI 输出审核与人工复核记录
- 诊断标签表
- 班级级反馈模板

## 4. 四个质量门

1. `目标门`
   本节目标必须能映射为任务、证据和评价。
2. `讲义门`
   讲义必须通过 C/M/A 与统一主线双重检查。
3. `设计门`
   作业设计必须同时产出教师端与学生端规则，不得只有教师稿。
4. `反馈门`
   批改结果必须能回写为班级高频问题与下一轮教学动作。

## 5. 教师不可外包的判断

1. 是否通过讲义
2. 是否采用某套作业模板
3. 红线问题是否成立
4. 哪些雷同样本进入纪律性处理
5. 哪些问题需要升级为课程系统资产

## 6. 当前最小运行策略

1. 先打通 `kernel -> skills -> runtime -> evaluation`
2. 先迁移母规则与模板，不急着全量迁移章节正文
3. 先以第二章、第三章等高频章节验证规则
4. 先生成章节变量配置，再决定章节资产如何迁移
