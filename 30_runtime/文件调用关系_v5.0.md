# 文件调用关系 v5.0

## 1. Concept

本文件用于解释 `projects5.0` 中各类文件的上下游关系。

它的目的不是重复列目录，而是帮助：

1. 教师
2. ChatGPT Projects
3. GitHub 连接场景下的模型

理解：

`哪些文件定义规则，哪些文件是 single source，哪些文件是正式产物，哪些文件只是治理与说明。`

如果没有这份关系说明，模型很容易把仓库理解成“很多平行文件”，从而出现：

1. 把 `assignment pack` 当成母规则
2. 把 `README` 当成 single source
3. 不知道讲义层当前采用 `B 方案`
4. 不知道评分输出如何回流到 `memory`

## 2. Mechanism

## 2.1 课程母规则层

### 作用

定义课程目标、评分原则、责任边界、运行流程。

### 代表文件

1. `README.md`
2. `00_kernel/系统总控与ai-skill-os映射_v5.0.md`
3. `00_kernel/课程项目化作业评估母规则_v5.0.md`
4. `30_runtime/12步运行手册_v5.0.md`
5. `30_runtime/ChatGPT_Projects项目指令词_v5.0.md`

### 调用关系

它们是全系统的上游解释层。  
章节层和产物层都不能违背这里的规则。

---

## 2.2 章节主配置层

### 作用

这是章节级 `single source of truth`。

### 代表文件

1. `30_runtime/chapter_master_configs/*.yaml`
2. `30_runtime/chapter_master_configs/README_v5.0.md`

### 调用关系

章节正式产物应优先从这里派生。  
如果章节主线、任务分值、关系图要求、专业校验点发生变化，应优先改这一层，而不是先改下游产物。

---

## 2.3 正式产物层

### 作用

把章节主配置转换成教学和评分可直接使用的正式文件。

### 代表文件

由 `chapter_master_configs/*.yaml` 派生：

1. `40_evaluation/runtime/chapter_profiles/*.json`
2. `50_assets/assignment_packs/<chapter_dir>/`
3. `50_assets/课堂实施方案/<chapter_dir>/lesson_script_input_v5.yaml`
4. `50_assets/课堂实施方案/<chapter_dir>/*课堂实施方案_v5.0.md`

### 调用关系

这一层是教学实施和评分运行的直接输入。  
它依赖章节主配置，但本身不是 single source。

---

## 2.4 讲义层

### 作用

承担教学表达、机制展开、课堂叙事与预习支撑。

### 代表文件

1. `50_assets/章节讲义/<chapter_dir>/`
2. `50_assets/章节讲义迁移策略_v5.0.md`
3. `50_assets/章节讲义/讲义层治理策略_v5.0.md`

### 当前治理方案

讲义层当前采用 `B 方案`：

`独立维护，但纳入 preflight 轻量一致性检查，不并入章节 single source。`

### 调用关系

讲义层应与章节主线、assignment pack、课堂实施方案保持一致，  
但不做正文级 drift check，也不由主配置逐字生成。

---

## 2.5 评分与反馈层

### 作用

把学生作业转成：

1. 分数
2. 评估 PDF
3. debug
4. 班级级教学反馈

### 代表文件

1. `40_evaluation/runtime/course_assignment_eval_v5.py`
2. `40_evaluation/runtime/build_teaching_feedback_v5.py`
3. `40_evaluation/班级级教学反馈报告模板_v5.0.md`

### 调用关系

这一层读取：

1. chapter profile
2. assignment pack
3. 作业目录
4. Excel / debug / PDF

它不应反向充当 single source，但它会生成可回流的数据。

---

## 2.6 memory 层

### 作用

吸收真实教学反馈，形成长期误区、案例和修订依据。

### 代表文件

1. `10_memory/README_v5.0.md`
2. `10_memory/误区库_v5.0.md`
3. `10_memory/案例库_v5.0.md`
4. `10_memory/班级教学反馈索引_v5.0.md`
5. `10_memory/runtime/memory_update_v5.py`

### 调用关系

这是反馈后的回流层：

`评分输出 -> 教学反馈 -> memory -> 修订讲义 / assignment pack / 课堂实施方案 / 评分规则`

---

## 2.7 运行与维护层

### 作用

保证全系统：

1. 可重建
2. 可检查
3. 可发布
4. 可维护

### 代表文件

1. `30_runtime/build_chapter_assets_v5.py`
2. `30_runtime/build_all_chapter_assets_v5.py`
3. `30_runtime/check_all_chapter_asset_drift_v5.py`
4. `30_runtime/maintain_projects5_v5.py`
5. `30_runtime/preflight_projects5_v5.py`
6. `30_runtime/preflight_local_v5.sh`
7. `Makefile`
8. `justfile`

### 调用关系

这一层不定义教学目标，而负责：

1. 从 single source 重建产物
2. 检查产物与主配置是否漂移
3. 对讲义层做轻量一致性门禁
4. 输出发布前摘要

## 3. Application

## 3.1 最小理解顺序

如果是教师或 ChatGPT Project 第一次进入系统，建议按以下顺序理解：

1. `README.md`
2. 课程母规则层
3. 章节主配置层
4. 正式产物层
5. 讲义层
6. 评分与反馈层
7. memory 层
8. 运行与维护层

## 3.2 最小判断规则

可用这组判断快速区分文件角色：

1. 若文件定义课程目标、评分原则、责任边界：
   - 属于 `课程母规则层`
2. 若文件定义某一章的核心结构变量：
   - 属于 `章节主配置层`
3. 若文件是根据章节配置派生出来的交付物：
   - 属于 `正式产物层`
4. 若文件用于教学展开但不纳入章节 single source：
   - 属于 `讲义层`
5. 若文件把作业转成分数、报告、debug：
   - 属于 `评分与反馈层`
6. 若文件把真实反馈沉淀为长期误区和案例：
   - 属于 `memory 层`
7. 若文件负责重建、检查、发布前门禁：
   - 属于 `运行与维护层`

## 3.3 一句话调用链

`课程母规则 -> 章节主配置 -> 正式产物 -> 教学实施 / 评分 -> 教学反馈 -> memory -> 回写修订`

## 4. 结论

在 `projects5.0` 中：

1. `chapter_master_configs/*.yaml` 是章节正式 single source
2. `assignment pack / lesson script / 课堂实施方案 / profile` 是正式产物
3. `章节讲义` 当前独立维护，但纳入轻量治理
4. `debug / 教学反馈 / memory` 才是系统闭环真正发生的地方

因此，任何使用本系统的人，都不应只看单个文件，而应按这条调用链理解整个工程。
