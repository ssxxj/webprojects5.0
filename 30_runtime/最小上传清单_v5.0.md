# 最小上传清单 v5.0

## 1. 概念

本清单用于控制每次进入 ChatGPT Project 或其他 AI 工作流时的最小上下文输入。

`v5.0` 的原则不是“上传得越多越好”，而是：

1. 上传足够支撑当前步骤
2. 避免重复资产污染上下文
3. 在 Step 11 场景下，确保 AI 能同时生成作业、评分和反馈链

## 2. 首次建项必传

1. `00_kernel/系统总控与ai-skill-os映射_v5.0.md`
2. `00_kernel/课程项目化作业评估母规则_v5.0.md`
3. `30_runtime/ChatGPT_Projects项目指令词_v5.0.md`
4. `30_runtime/12步运行手册_v5.0.md`
5. `40_evaluation/教学评估与证据标准总表_v5.0.md`
6. `10_memory/README_v5.0.md`
7. `50_assets/README_v5.0.md`

这些文件足以建立当前 5.0 的课程级口径：

- 课程目标如何落到作业
- 红线与维度扣分如何分离
- Step 11 要生成哪些正式产物

## 2A. 若已连接 GitHub 仓库

若 `projects5.0` 已完整上传到 GitHub，且 ChatGPT Project 已连接该仓库，则不建议再手动上传整套章节产物。

此时建议只保留以下“稳定入口文件”：

1. `README.md`
2. `30_runtime/ChatGPT_Projects项目指令词_v5.0.md`
3. `30_runtime/12步运行手册_v5.0.md`
4. `30_runtime/最小上传清单_v5.0.md`
5. `30_runtime/chapter_master_configs/README_v5.0.md`
6. `50_assets/章节讲义/讲义层治理策略_v5.0.md`

若需要 GitHub 连接模式的完整说明，优先查看：

7. `30_runtime/GitHub连接版ChatGPT_Projects启动包_v5.0.md`
8. `30_runtime/文件调用关系_v5.0.md`

## 3. 进入 Step 11 时必须补传

若当前任务是布置作业、生成项目、设计评分或生成学生提交说明，必须再补传：

1. `20_skills/SK07_课后练习与章节项目生成器_v5.0.md`
2. `40_evaluation/第11步作业设计模板_v5.0.md`
3. `40_evaluation/学生版作业说明单模板_v5.0.md`
4. `40_evaluation/教师验收表模板_v5.0.md`
5. `40_evaluation/章节作业变量配置模板_v5.0.md`
6. `40_evaluation/班级级教学反馈报告模板_v5.0.md`

如果需要同时做评分脚本或 profile 设计，再补传：

7. `40_evaluation/runtime/course_assignment_eval_v5.py`

## 4. 进入章节评分或复核时必须补传

若当前任务是重评分、规则核查、人工复核或批量输出评估结果，必须补传：

1. 当前章节作业说明或 Notion/PDF 要求
2. 当前章节变量配置
3. 当前评分输出样本
4. 如已存在，再补传本章 debug、Excel、评估 PDF

## 5. 进入迁移或系统升级时建议补传

若当前任务是从 4.0 迁移到 5.0，建议再补传：

1. 对应的 `projects4.0` 同名文件
2. 本轮迁移目标说明
3. 上一轮复盘记录

## 6. 不建议上传

- 已有合并版后，再重复上传大量旧版草稿
- 同一章节的多个过期中间文件
- 与当前步骤无关的全量学生作业包
- 未经筛选的重复 debug 或导出文件

## 7. 应用

实际操作时可用这条最小判断：

- 只做章节讲义：上传讲义链
- 只做 Step 11：上传 Step 11 设计链
- 只做评分：上传 rubric + 章节变量 + 评分脚本链
- 做 4.0 -> 5.0 升级：上传同名规则文件对照链

若使用 GitHub 连接模式，则改为：

- 只保留入口文件在 Project 中常驻
- 当前章节相关文件通过 GitHub 仓库按需读取
- 仅在需要聚焦单一章节时，再额外点名该章节的主配置和正式产物
- 若模型容易把文件看成平行文档，补传 `30_runtime/文件调用关系_v5.0.md`
