# 12 步运行手册 v5.0

## 1. 适用范围

- 高职信息安全专业 0 基础学生
- DVWA 或教师授权环境
- 以章节项目和课后作业为主要反馈载体

## 2. 90 分钟标准节奏

- 10 分钟：目标、边界、场景
- 20 分钟：Concept
- 25 分钟：Mechanism
- 25 分钟：Application
- 10 分钟：形成性评估、作业布置、收束

## 3. 12 步与核心产出

| Step | 输入 | 输出 | 主 skill | 通过条件 |
| --- | --- | --- | --- | --- |
| 1 | 课程定位 | 学期目标与边界 | `SK02` | 可执行 |
| 2 | 学生起点 | 前测与补救 | `SK02` | 可分层 |
| 3 | 学期目标 | 章节里程碑 | `SK03` | 一章一问题 |
| 4 | 章节目标 | 单元目标与证据 | `SK03` | 一目标一证据 |
| 5 | 章节目标 | 初稿 | `SK04` | 可加工 |
| 6 | 初稿 | 教师版讲义 | `SK04` | 可授课 |
| 7 | 讲义 | 双评结果 | `SK05` | 达到通过线 |
| 8 | 已通过讲义 | 授课实施方案 | `SK06` | 90 分钟可执行 |
| 9 | 课堂表现 | 观察记录 | `SK08` | 有最小观察记录 |
| 10 | 观察记录、作业结果 | 复盘与归因 | `SK08` | 有改进行动 |
| 11 | 单元目标或章节目标 | 教师端作业包 + 学生版说明单 + 分值拆分 + 红线表 + 诊断标签 + 班级反馈模板 | `SK07` | 可提交、可评分、可诊断、可反馈 |
| 12 | 章节数据 | 资产沉淀与流程升级 | `SK08` | 有可复用资产 |

## 5. Step 11 自动化入口

若本章已经进入 `projects5.0` 的单一来源模式，优先编辑：

- `30_runtime/chapter_master_configs/<chapter_id>.yaml`

再统一编排：

```bash
python3 30_runtime/build_chapter_assets_v5.py \
  --input 30_runtime/chapter_master_configs/<chapter_id>.yaml
```

这会一次性回写：

1. `chapter profile`
2. `assignment pack`
3. `lesson_script_input_v5.yaml`
4. `课堂实施方案`

只有在临时试验或迁移过渡期，才建议直接分别运行 Step 8 / Step 11 的单独生成器。

若需一次性重建全课程章节资产，可运行：

```bash
python3 30_runtime/build_all_chapter_assets_v5.py
```

默认批量处理：

- `30_runtime/chapter_master_configs/*.yaml`

并输出：

- `30_runtime/chapter_master_configs/all_chapters_build_manifest_v5.json`

若需检查当前落盘产物是否已与主配置漂移，可运行：

```bash
python3 30_runtime/check_all_chapter_asset_drift_v5.py
```

默认输出：

- `30_runtime/chapter_master_configs/all_chapters_drift_report_v5.json`

若要把“批量重建 + drift check”收束成一个课程维护命令，默认使用：

```bash
python3 30_runtime/maintain_projects5_v5.py
```

模式说明：

1. `--mode full`
   先重建，再检查漂移；适合日常维护。
2. `--mode check`
   只做漂移检查；适合 CI。
3. `--mode rebuild`
   只重建，不做漂移检查。
4. `--mode validate`
   只校验主配置，不写产物。

统一报告默认写到：

- `30_runtime/chapter_master_configs/projects5_maintenance_report_v5.json`

若只需要发布前高层摘要，不想展开整份详细 JSON，使用：

```bash
python3 30_runtime/preflight_projects5_v5.py
```

它只输出：

1. `release_ready`
2. 主配置校验是否通过
3. 是否存在 drift
4. 失败章节列表

默认摘要写到：

- `30_runtime/chapter_master_configs/projects5_preflight_report_v5.json`

本地日常入口：

```bash
make test
make maintain
make preflight
make check
```

或使用：

```bash
just test
just maintain
just preflight
just check
```

本地 CI 入口：

```bash
make ci-local
```

GitHub Actions 工作流入口：

- `.github/workflows/projects5-v5-ci.yml`

当章节 profile 已存在时，可直接运行：

```bash
python3 40_evaluation/runtime/generate_assignment_pack_v5.py \
  --profile 40_evaluation/runtime/chapter_profiles/<chapter_profile>.json \
  --outdir <输出目录>
```

默认会生成：

1. 教师端作业包
2. 学生版作业说明单
3. 教师验收表
4. 班级级教学反馈模板
5. 任务分值与红线摘要
6. 生成清单 manifest

若作为正式章节资产保存，建议输出到：

`50_assets/assignment_packs/<chapter_dir>/`

已有章节作业包入口见：

- `50_assets/assignment_packs/README_v5.0.md`
- `50_assets/assignment_packs/第4到第8章_4.0讲义到5.0作业包对照表.md`

## 6A. Step 8 授课实施入口

Step 8 的目标不是“写一份教案”，而是完成：

`章节讲义 -> 课堂动作 -> 形成性评估 -> 作业收口 -> 观察记录入口`

最小输入建议：

1. 教师版章节讲义
2. 对应 assignment pack
3. 对应 chapter profile
4. 当前班级基础判断
5. 当前环境与授权边界

若已存在历史 memory，Step 8 默认还应查看：

1. `10_memory/误区库_v5.0.md`
2. `10_memory/案例库_v5.0.md`
3. `10_memory/班级教学反馈索引_v5.0.md`

Step 8 默认优先调用：

- `20_skills/SK06_授课实施设计器_v5.0.md`
- `50_assets/课堂实施方案模板_v5.0.md`
- `50_assets/课堂实施方案/逐章授课脚本生成器输入规范_v5.0.md`
- `50_assets/课堂实施方案/逐章授课脚本输入模板_v5.0.yaml`

如果后续进入“逐章授课脚本生成器”自动化阶段，Step 8 推荐顺序改为：

1. 先用 `逐章授课脚本输入模板_v5.0.yaml` 整理章节变量
2. 再根据 `逐章授课脚本生成器输入规范_v5.0.md` 校验输入完整性
3. 最后交给授课脚本生成器或 SK06 产出正式课堂实施方案

自动化入口：

```bash
python3 50_assets/课堂实施方案/runtime/generate_lesson_script_v5.py \
  --input 50_assets/课堂实施方案/逐章授课脚本输入模板_v5.0.yaml
```

如果已经为多个章节准备好 `lesson_script_input_v5.yaml`，可直接批量生成：

```bash
python3 50_assets/课堂实施方案/runtime/generate_all_lesson_scripts_v5.py
```

如果这些输入来自章节主配置，则应优先通过：

```bash
python3 30_runtime/build_chapter_assets_v5.py \
  --input 30_runtime/chapter_master_configs/<chapter_id>.yaml
```

让 `lesson_script_input_v5.yaml` 与 `chapter profile / assignment pack` 保持同源。

通过条件：

- [ ] 只有 1 条本节唯一主线
- [ ] 至少 1 个“先判断再验证”的课堂动作
- [ ] 至少 1 个形成性评估点
- [ ] 至少 1 个 assignment pack 对齐收口动作
- [ ] 至少保留 3 个 Step 9 观察项

## 7. Step 9 最小观察记录

进入 `SK08` 前，课堂观察至少要保留以下 5 项中的 3 项：

- [ ] 本节目标是否完成
- [ ] 学生最常见的卡点是什么
- [ ] 哪个环节耗时异常
- [ ] 哪类错误是讲义问题而不是学生问题
- [ ] 哪个样本可作为正例或反例

如果只剩教师印象、没有结构化记录，也要产出“最小观察记录”，但必须在复盘里注明：

- 观察数据不足
- 本次结论只作为临时判断
- 下次要补哪些记录

## 8. Step 10 复盘入口

Step 10 的目标不是“写总结”，而是完成：

`观察 -> 归因 -> 改进行动 -> 系统升级建议`

最小输入建议：

1. 课堂观察记录
2. 至少 1 份正例作业
3. 至少 1 份待补救样本
4. 班级评分 debug JSON
5. 对应章节 profile

复盘时优先区分 6 类来源：

1. 讲义解释不足
2. 作业设计不清
3. 评分口径不稳
4. 前置知识缺口
5. 环境与组织问题
6. 安全/合规问题

若班级 debug 已存在，Step 10 默认优先调用：

- `20_skills/SK08_课堂观察复盘与流程升级器_v5.0.md`
- `40_evaluation/runtime/build_teaching_feedback_v5.py`

通过条件：

- [ ] 至少形成 3 条可执行改进行动
- [ ] 至少指出 1 个需要升级的系统文件
- [ ] 不把所有问题都归因给学生

## 9. Step 12 资产升级判断

Step 12 只处理“值得沉淀”的改动，不把一次性问题都写进系统。

优先升级到资产或规则层的情况：

1. 同类问题连续两次以上出现
2. 同一标签在一个班中高频出现
3. 作业或评分误判会重复发生
4. 讲义和作业目标不一致
5. 学生反复不会审核 AI 输出

输出落点建议：

- 讲义解释问题：回写章节讲义或讲义模板
- 作业结构问题：回写 `SK07`、作业包或学生版说明单
- 评分口径问题：回写 `chapter_profiles` 或评分引擎
- 复盘与反馈问题：回写 `SK08`、教学反馈模板或 runtime 脚本
- 可迁移案例：进入 `10_memory` 或 `90_growth`

## 10. 班级反馈自动化入口

当批量评分已经生成 debug JSON 后，可直接运行：

```bash
python3 40_evaluation/runtime/build_teaching_feedback_v5.py \
  --debug <debug.json> \
  --profile 40_evaluation/runtime/chapter_profiles/<chapter_profile>.json
```

默认会生成：

1. 教学反馈报告 Markdown
2. 教学反馈报告 PDF
3. 教学反馈摘要 JSON

典型用途：

1. 输出班级平均分与已提交/未提交情况
2. 汇总各维度平均分
3. 汇总高频错误标签与高风险问题
4. 给出下轮教学动作和系统文件升级建议

若已生成班级反馈，则 Step 10 和 Step 12 的最小动作应当是：

1. 先判断哪些问题属于下次课要补讲
2. 再判断哪些问题需要升级模板、profile 或脚本
3. 最后决定哪些内容进入资产沉淀
